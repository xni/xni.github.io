Competition: C++ delayed call
2020-06-05 13:30

Amount of positive feedback foforious post motivated me to go one step further to put a fat dot on this topic.

Let’s have some fun! I am launching a competition for the fastest code for making a delayed function call.

The rules are:

  1. You are sending me your solution before 14th of June 2020 to smartlemon@protonmail.com
  2. I’m running some tests for correctness. You don’t know my tests. I will reply back to you, if the tests are failing, describing the error. Each submission which fails test adds you 1 yellow card.
  3. Amongs all of the correct submissions (passing every test), sent to me before 2020-06-14 23:59:59 UTC I will select the fastest one using a pretty trivial set of cases. I will select the fastest by creating a benchmark on quick-bench. If there are multiple fastest solutions, one with the least amount of yellow cards will win.
  4. The winner will receive my congratulations and respect from other participants.

Now about the task. You need to use C++17. You need to write a templated C++ class, MaybeDelayedCall. Constructor accepts at least 2 arguments, the argument arg to pass to the function F, and a boolean param now. Function F, which needs to be called can be passed as an argument as well, or using auto template parameter. If you are using auto template parameter, you are allowed to have one failed submission without a yellow card. If you are using C++17 template type auto deduction for classes, you are allowed to have one failed submission without a yellow card. If now param is true, function F should be called with arg immediately. If now param is false, MaybeDelayedCall should store the argument (in other words, if arg was passed as a reference, you are not allowed to save that reference, because our delayed call can not guarantee lifetime expectations for the agrument), and call function F only when the method OnTime() of MaybeDelayedCall is invoked. It is guaranteed that in every test, where I’m going to do MaybeDelayedCall<...> x(arg, now), it is correct to write F(arg), in other words, MaybeDelayedCall should not deal with a problem that arg is not compatible with F.

Also important note: the performance tests will have a following structure

static void TestForKostya()
{
    for (int i = 0; i < 10000; ++i) {
        MaybeDelayedCall<...> x(arg, now);
        x.OnTime();
    }
}

so you don’t have to deal with delayed calls, we are just measuring the effeciency of a storing/passing. Also important note - object x is allocated on the stack.

If you have any questions, don’t hesitate to reach me by email smartlemon@protonmail.com.

Good luck!
