  2. Guest 6 months ago

    This case is addressed through the use of git-flow's support branches (which, unfortunately, are not very well-documented yet).

    If you want to support legacy releases, use

    `git flow support start <release> <base>`

    where `<base>` must be a commit on master. This essentially creates a separate "production" branch for that release, where you can apply hotfixes and update minor version numbers without affecting the mainline release.

          I think I have an example that can show the benefits. Consider this example:
          Developer 1 writes code for feature A then commits & pushes.
          Developer 2 writes code for feature B then commits & pushes.
          Developer 1 writes more code for feature A (say, integration testing came back with a major bug) then commits & pushes. Feature A is now complete.

          Now the customer wants a release. Feature A is ready but feature B isn't. I *could* wait for developer 2 to get his act together and finish feature B or I *could* release the buggy version of feature A but I have paying customers to support so instead I will have to cherry pick the commits for feature A so that buggy feature B doesn't get into the release. I'll use a branch for this since it will result in a merge. I can't merge this back to my develop branch since I don't actually want to remove feature B. So I'll create a new branch, let's call this branch "release" for now. As a bonus I can refer to this branch later if my paying customer needs a hotfix. I'll also tag it with the release. So far so good, and this sounds like what you suggested.

I could branch off of develop for every release but then I'll have to do that nasty feature A cherry pick again and it would be easier to just cherry pick new feature commits to my existing release branch. So I'll continue using this branch and just tagging new releases as they are ready. If you know you're going to use this model ahead of time you may as well use master for this release branch.

I get that fast-forwarding is the best-case scenario but it doesn't happen as often as we would like in the real world with multiple developers working on separate distinct features with different timelines, etc...

    How does this work with a shared test-suite that all feature branches need to run to ensure no regression has been made?

    A nice approach, but this seems to work best for features that are not functionally related.

    In my experience with git and feature branching, the main stopper for using a "feature branching" git workflow has been TDD concurrency with our continuous-integration-style developer flow.

    Team members working on feature B need to know several times a day that they are not treading on feature A's toes. This means that you are performing a pull --rebase, test, and push throughout the day. The idea being that frequent commits will bring to light testing failures early.

    I've tried a couple times to design a git workflow gem/node module around feature branches that could "synchronize" with a common develop branch that testing would be performed upon while keeping the git history of different features separate and distinguishable.

    The best that I could come up with supporting this need to stay in sync between features while maintaining branch histories was to continuously apply squasing/patching to a common branch (say, develop) while rebasing/merging changes from others into your own feature branch.

    If you don't have a stable-member, long-lived team on which all developers can incorporate their workflow together you can constant exceptions to the rule. New devs "go vanilla" and done use the gem, don't understand the workflow, or just resort to "trunk" development without feature branching.

    It has almost always come down to "trunk" development with large, enterprise teams that are working on inter-related features in parallel. It makes tracking through changes a headache, but at least then you can enforce some sort of commit-template update-hook on the "central repository" that makes logging changes and developer additions somewhat structured.

  65. Gabe da Silveira over 4 years ago

    I've tagged the original article to read more thoroughly. It's definitely an interesting workflow that I want to study, but it's in conflict with my current preferred workflow which is rebase heavy.

    In general I prefer rebasing topic branches rather than merging them. I've come to this view over the course of working on a 3-year project in git. I think a no-rebase policy is good for learning git, but once you understand the fundamentals and how rebase works then there are more subtle questions to consider.

    The assertion that rebasing loses information that merging preserves is too simplistic. Yes, you lose information about the original development history. However you also overload conflict resolution into single, possibly huge merge commits, whereas with rebase the conflict resolutions are done on a very granular level as if the original developer had had to resolve them as they made each commit. Granted this is possibly more work and not necessarily worth the effort—it's definitely a tradeoff to be considered—however there is one common case where a rebase workflow gives you a huge win: logical but not literal conflicts. That is to say, if two branches introducing mutually breaking changes that don't happen to conflict (ie. they don't change nearby lines, but nevertheless they are incompatible changes). If those two branches are merged then git-bisect will just give you the (possibly gargantuan) merge commit. However if one is rebased on top of the other you will get a single granular commit.

    I've written about this at http://darwinweb.net/articles/...
