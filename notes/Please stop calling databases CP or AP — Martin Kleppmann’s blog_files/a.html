          Coda Hale • 8 days ago

          “However, if you require quorum reads and writes (R+W>N), and you have a network partition, clients on the minority side of the partition cannot reach a quorum, so quorum operations are not CAP-available (at least temporarily, until the database sets up additional replicas on the minority side).”

          Cassandra, Riak, and Voldemort all implement what they call hinted handoff and what the Dynamo paper refers to as “sloppy quorum”. If they can’t achieve a quorate write, they will write to the coordinator node (or a random reachable node) which will then propagate the write when the cluster is no longer partitioned. Depending on the system and configuration, this may reduce to W=1.

                                        Imad Qureshi Martin Kleppmann • 18 hours ago

                                        Well, both HBase and Google's Big Table guarantee linearizability by making sure that request for a particular row key always goes to the same server. Two clients asking a question will always get the same answer or no answer at all. Data is distributed across the cluster based on row key. In case of node failure, the data served by the failed server is then served by other nodes who have a copy (Hadoop makes sure three copies of data). By strongly consistent, I meant it is consistent as in CP.

                                        Master in HBase is of less significance than other systems. On first call, client needs to connect to HMaster to find out which server is hosting the key, but then it can client directly goes to the server hosting the key. The only time this client will need to go to Master again is, if the server has failed or there is a split causing the key to move to a different server.
